Portfolioaufgabe Java Enterprise Edition

Wow! Jetzt haben wir so viele Dinge rund um die Java Enterprise Edition gesehen, dass es langsam Zeit wird, sich anderen Themen zuzuwenden. 🙂 Doch zuvor dürft ihr noch einmal eine komplette Webanwendung entwickeln, mit allem was dazu gehört: Persistence Entities, Benutzerverwaltung, Enterprise Java Beans, Servlets und Java Server Pages. 🎭 Als Grundlage verwenden wir dafür die jTodo-Anwendung aus der benoteten Hausaufgabe und bauen sie so um, dass sie keine Aufgaben mehr verwaltet, sondern Verkaufsangebote und Gesuche. 💰

Folgende Anforderungen soll die neue Anwendung erfüllen:

Allgemeines

  • Natürlich heißt die Anwendung nicht mehr jTodo, sondern irgendwie anders. 🙂
  • Die Umbennung soll den Namen des Projekts, die Javapakete, die Datenbanktabellen, das Datenbankschema in der Datei Other Sources/META-INF/orm.xml sowie den Namen der Benutzerrolle in der Web Pages/WEB-INF/web.xml umfassen.
  • Das Hintergrundbild solltet ihr auch wechseln. 🏖️

Benutzerprofil

  • Zu jedem Benutzer sollen folgende Zusatzwerte gespeichert werden: Name, Anschrift, PLZ, Ort, Telefon, E-Mail
  • Diese Daten sollen bereits bei der Registrierung eingegeben werden.
  • Zusätzlich soll es für angemeldete Benutzer die Möglichkeit geben, die eigenen Daten zu bearbeiten.
  • Die in den Screenshots dargestellte Möglichkeit, sein Passwort zu ändern, müsst ihr nicht realisieren.

Angebote und Gesuche

  • Anstelle von Aufgaben sollen Verkaufsanzeigen verwaltet werden.
  • Folgende Felder soll eine Anzeige beinhalten: Ersteller, Kategorie, Art (Biete oder Suche), Titel, Beschreibung, Erstelldatum, Preis, Art des Preises (Festpreis oder Verhandlungsbasis)
  • Auf der Seite einer Anzeige sollen auch Name und Adresse des Erstellers angezeigt werden.
  • Die generelle Logik zur Suche, Auflistung und Bearbeitung der Anzeigen soll beibehalten werden.
  • Jedoch darf eine Anzeige nur von dem Benutzer bearbeitet werden, der sie auch erstellt hat. Für alle anderen Benutzer sollen die Felder read-only sein und POST-Anfragen dürfen nicht zu einer Änderung der Daten führen.

Hinweise
  • Ladet auch für diese Aufgabe die allerneuste Version von jTodo aus Moodle.
  • Wenn ihr den Namen der Benutzerrolle ändert, vergesst nicht die Annotationen @RolesAllowed der Beans, die Methode signup() in der UserBean sowie die Datei glassfish-web.xml anzupassen. Sonst kommt es beim Login zu folgendem Fehler: Access to the specified resource has been forbidden.
  • Nach größeren Umbauten kann es nötig sein, einmal Clean and Build auszuführen, bevor die Anwendung wieder gestartet werden kann.
  • Nach dem Umbenennen der Pakete müsst ihr die <jsp:useBean>-Anweisung in der task_list.jsp anpassen.
  • Die Methode getCurrentUser() der UserBean liefert den Stammsatz des eingeloggten Benutzers zurück.
  • Die E-Mailadresse kann mit @Pattern(regexp = "^\\w+@\\w+\\..{2,3}(.{2,3})?$") validiert werden.
  • Das aktuelle Datum kann mit new Date(System.currentTimeMillis()) ermittelt werden.
  • Die aktuelle Uhrzeit kann mit new Time(System.currentTimeMillis()) ermittelt werden.
  • Ein Eingabefeld kann mit <input … ${readonly ? 'readonly="readonly"' : ''} … > so gestaltet werden, dass es über die Variable readonly nicht-eingabebereit geschaltet werden kann.
  • Wenn ihr das Layout der Seite anpassen wollt, müsst ihr die Dateien im Order Web Pages/less anstelle der CSS-Dateien bearbeiten.
